# Tongue Diagnosis POC — Design

**Date:** 2026-05-06
**Status:** Implemented via PRs #1 (`worktree-feat-tongue-poc`) and #2 (`refactor/typed-api-responses`).
**Partially superseded by:** [`2026-05-07-merge-amanda-resnet50-design.md`](./2026-05-07-merge-amanda-resnet50-design.md) (PR #4, merge SHA `473e895`). The 11-per-task-ONNX-head architecture in §3 and §6 was replaced with two PyTorch composite heads served from Hugging Face Hub. ONNX support remains as a sibling backend, but the default `registry.default.yaml` no longer ships ONNX heads. See the 2026-05-07 spec for the post-merge architecture.
**Stack:** Gradio (frontend) · FastAPI (backend) · ONNX Runtime + PyTorch (inference) · Google ADK → Gemini (LLM)

---

## 1. Goal

A small end-to-end proof of concept that lets a user:

1. Capture or upload a tongue image (webcam or file).
2. Run per-task ONNX classifiers covering the 11 active categories of the v4 TCM tongue-inspection schema.
3. Render the per-head labels (with confidences) and a plain-language doctor comment generated by Gemini under a locked rule-bound prompt.
4. Tune **all** runtime configuration (system prompt, LLM parameters, model registry) from the same UI without redeploying.

The POC is single-user, local-first, and deliberately small — every decision below favours minimum moving parts over future-proofing.

## 2. Non-goals

- Multi-user / authentication / authorisation.
- Persistent session history or audit logs.
- Real-time streaming of partial results (single sync `/api/analyze` is enough; SSE is a follow-up).
- Mobile / native packaging.
- Accuracy targets, evaluation harness, or model training pipelines (models are pre-built ONNX artefacts).
- Multimodal LLM fallback — Gemini receives **only** classifier labels per the locked rules.

## 3. Architecture

```
┌──────────────────────┐  HTTP multipart   ┌──────────────────────────────────────┐
│  Gradio frontend     │ ─────────────────►│  FastAPI backend                     │
│  (tongue-frontend)   │   POST /api/      │  (tongue-backend)                    │
│                      │     analyze       │                                       │
│  • webcam capture    │                   │   1. ROI detection (optional ONNX)    │
│  • file upload       │ ◄─────────────────│   2. Per-task ONNX inference loop     │
│  • result panel      │   JSON {heads,    │   3. Build zh-TW user message         │
│  • prompt editor     │   comment, ...}   │   4. Read prompt + LLM config         │
│  • registry editor   │                   │   5. ADK → Gemini → comment           │
└──────────────────────┘                   └──────────────────────────────────────┘
                                                        │             │
                                                        ▼             ▼
                                         ┌──────────────────┐ ┌────────────────┐
                                         │ tongue-ai        │ │ Google ADK SDK │
                                         │ (PyTorch + ORT)  │ │ → Gemini       │
                                         └──────────────────┘ └────────────────┘
```

**Three boundaries, three responsibilities:**

- `tongue-frontend` (Gradio) — capture + display + config editing only. No inference, no Gemini calls. HTTP-only contact with backend.
- `tongue-backend` (FastAPI) — orchestration. Owns the pipeline (ROI → classify → assemble prompt → call LLM → return) and the persisted "current" config files.
- `tongue-ai` (PyTorch + ONNX Runtime) — pure inference library. Model registry, preprocessing, ONNX session management. No web, no LLM. Importable for tests and future batch eval.

The existing repo already enforces "no Python imports across frontend/backend"; this design preserves that. ADK lives in `tongue-backend`, not `tongue-ai`, so the AI package can be unit-tested without network.

## 5. Core abstractions

### `tongue_ai.task_head.TaskHead`

```python
class TaskHead:
    name: str                             # "舌色"
    head_type: Literal["single", "multi"]
    session: ort.InferenceSession
    input_size: tuple[int, int]           # (224, 224)
    normalise: Normalisation              # mean/std
    class_names: list[str]                # ["淡","淡紅","微紅","紅","絳","青紫","暗"]
    threshold: float = 0.5                # multi-label only

    def predict(self, image_bgr: np.ndarray) -> HeadResult: ...
```

One ONNX model + everything required to use it. Adding a new schema head = drop a `.onnx` and add a registry entry. Zero code change.

### `tongue_ai.types.HeadResult`

```python
@dataclass
class ClassScore:
    label: str                            # "淡紅"
    score: float                          # 0.78

@dataclass
class HeadResult:
    task: str                             # "舌色"
    head_type: Literal["single", "multi"]
    predictions: list[ClassScore]         # len=1 (single) | ≥1 (multi above threshold)
    error: str | None = None              # set if predict() failed; predictions empty
```

### `tongue_backend.pipeline.analyze`

```python
def analyze(image_bytes: bytes) -> AnalyzeResponse:
    image    = decode_bgr(image_bytes)
    roi      = detect_tongue(image, registry.detector) or image
    heads    = inference.run_all(roi, registry)
    user_msg = user_message.build(heads)
    system   = prompt_store.load_current()
    llm_cfg  = config_store.load_llm()
    comment  = llm.client.run(system, user_msg, llm_cfg)
    return AnalyzeResponse(
        user_message=user_msg,
        heads=heads,
        comment=comment,
        disclaimer=DISCLAIMER,
        timing_ms=timing.snapshot(),
    )
```

The only place that knows the full flow. Routes stay thin parse/serialise wrappers.

## 6. Config plane

All runtime configuration is editable from the frontend. Three sections, each with shipped default + editable current:

| Section | Storage path (current) | Reload cost |
|---|---|---|
| **prompt** | `packages/backend/prompts/system.current.md` | free — re-read every analyze |
| **llm** | `packages/backend/config/llm.current.yaml` | free — re-read every analyze |
| **registry** | `packages/backend/config/registry.current.yaml` | expensive — explicit `Apply & Reload Models` |

The registry YAML lives under `tongue-backend/config/` (not `tongue-ai/`) because it describes how the backend orchestrates the AI package. `tongue-ai` exposes `load_registry(path)` and accepts whatever YAML the caller provides; the package itself ships only ONNX artefacts. ONNX paths inside the YAML resolve relative to the YAML's own directory, so they reach into `../../ai/models/`.

Defaults are shipped read-only. On first boot, current is copied from default if missing. Reset = overwrite current with default.

### API

```
GET    /api/config/{section}               → {content, is_default, mtime}
PUT    /api/config/{section}               ← {content}     # validates, persists
POST   /api/config/{section}/reset         → restores from default
POST   /api/config/registry/reload         → re-inits ONNX sessions in-process
```

`{section}` ∈ `prompt`, `llm`, `registry`.

`PUT /api/config/registry` only persists the YAML; the running registry remains unchanged until `POST /api/config/registry/reload` is called. This two-step is intentional: a typo in YAML never crashes the running server.

### Frontend tabs

Four tabs — each config section gets its own tab with its own Save / Reset / Reload buttons, so there's no ambiguity about which section a click targets.

```
┌──────────────────────────────────────────────────────────────────┐
│  舌診分析   │   提示詞設定   │   LLM 設定   │   模型設定           │
├──────────────────────────────────────────────────────────────────┤
│  Tab 1: capture (webcam / upload) + result panel                  │
│  Tab 2: [textarea: system prompt]                                 │
│         [Save] [Reset to default] [Reload from disk]              │
│  Tab 3: [textarea: LLM yaml — model, temperature, max_tokens, …]  │
│         [Save] [Reset to default] [Reload from disk]              │
│  Tab 4: [textarea: registry YAML]                                 │
│         [Save] [Reset] [Reload from disk] [Apply & Reload Models] │
│         status: "11 heads loaded · last reload 12:04:33"          │
└──────────────────────────────────────────────────────────────────┘
```

All editors are **raw text/YAML**. Validation is server-side on `PUT`; errors render in a small panel above the buttons. Structured forms can be layered on later without backend changes.

The four buttons mean four distinct things:

- **Save** — `PUT` the textarea content; persists to disk; for prompt/LLM applies on next analyze, for registry stays staged until `Apply & Reload Models`.
- **Reset to default** — `POST /reset`, then re-fetch; restores the shipped default into "current".
- **Reload from disk** — re-fetch only; discards the user's unsaved textarea edits.
- **Apply & Reload Models** (registry only) — `POST /reload`; re-inits ONNX sessions in the running process.

### Validation

| Section | Server-side check on `PUT` |
|---|---|
| prompt | None — anything is accepted (user owns the format consequence). |
| llm | YAML parses; `temperature` ∈ [0, 2]; `max_tokens` > 0; `model` non-empty string. |
| registry | YAML parses; required keys per task; each `onnx_path` exists on disk; `class_names` non-empty. ONNX is **not** loaded yet — that happens on `Reload`. |

## 7. Data flow

### `POST /api/analyze` request

Multipart with one field `file` (jpg/png, ≤ 10 MB).

### `POST /api/analyze` response shape

```json
{
  "user_message": "本次舌診判讀結果：\n- 舌色：淡紅（0.78）\n- 舌質：齒痕（0.84）、嫩（0.71）\n...",
  "heads": [
    {"task":"舌色","head_type":"single","predictions":[{"label":"淡紅","score":0.78}]},
    {"task":"舌質","head_type":"multi", "predictions":[{"label":"齒痕","score":0.84},{"label":"嫩","score":0.71}]}
  ],
  "comment": "## 主要中醫體質\n…\n\n## 次要中醫體質\n…\n\n## 體質說明\n…\n\n## 證素列表\n氣虛、痰濕\n",
  "disclaimer": "此為AI自動生成，不具醫療建議。若有疾病或疑問，應向專業中醫師諮詢。",
  "timing_ms": { "detect": 25, "infer": 130, "llm": 1850, "total": 2050 }
}
```

`user_message` is included in the response so the frontend can show "what we sent to Gemini" in a collapsed panel — invaluable while iterating on the prompt. `timing_ms` is POC-grade observability.

### Two-message Gemini prompt structure

- **System message**: the locked 大眾版 prompt (current contents of `system.current.md`). Includes the lookup table, 9 allowed 證素, specificity weights, pathogenesis chains, and required output format.
- **User message**: built per request from the `HeadResult` list, in zh-TW with confidences:

  ```
  本次舌診判讀結果：

  - 舌色：淡紅（0.78）
  - 舌質：齒痕（0.84）、嫩（0.71）
  - 舌態：無異常（0.91）
  - 朱點部位：無（0.95）
  - 瘀斑瘀點部位：無（0.93）
  - 裂紋：無異常（0.88）
  - 舌津：正常（0.80）
  - 舌苔顏色：白（0.86）
  - 舌苔厚薄：薄（0.74）
  - 舌苔根：有根（0.82）
  - 舌下絡脈：無異常（0.77）

  請依規則輸出大眾版報告。
  ```

  Single-label heads → one class. Multi-label heads → all classes above threshold, each with score. Heads with `error` are silently omitted.

### Frontend rendering (Tab 1)

Heads displayed as a small dataframe (task | predictions). Doctor comment rendered as **raw markdown** — the user controls the format via the editable prompt, so the frontend deliberately does not parse `主要體質 / 次要體質 / …` as structured fields. Disclaimer rendered as a small fixed footer.

A collapsed "進階" section shows `user_message` + `timing_ms` for debugging.

### Invariants

- `/api/analyze` re-reads prompt + LLM config every request. No caching layer.
- ONNX sessions are a process-level singleton held in `app.state.registry`. `POST /reload` swaps it atomically.
- The image is never sent to Gemini. Only the formatted user message. (Per locked rules: 對應表為唯一依據.)
- Heads that fail at inference don't kill the request. That head's entry becomes `{..., "error": "..."}`; user message excludes it; the rest proceeds.

## 8. Error handling & edge cases

### Input errors (4xx, user-fixable)

| Case | Response | Frontend behaviour |
|---|---|---|
| No file in multipart | `400 {error:"missing file"}` | Inline "請選擇或拍攝照片" |
| Decode fails (corrupt, unsupported) | `400 {error:"could not decode image"}` | Inline error |
| Image > 10 MB | `413 {error:"image too large"}` | Inline error |
| Webcam permission denied | (frontend-only — no backend call) | Show browser permission hint |

### Inference errors (graceful degradation)

| Case | Behaviour |
|---|---|
| ROI detector throws | Log + fall back to whole image; set `roi_used=false` in response |
| One TaskHead `predict()` raises | Replace that head's entry with `error`; user message simply omits that bullet; rest of pipeline continues |
| All heads fail | Still call Gemini — user message becomes `本次舌診判讀結果：（無可用判讀資料）`; locked prompt's rules already handle "no data" |
| ORT session corrupt | Detected at reload time (see Lifecycle below) |

### LLM errors

| Case | Response | Notes |
|---|---|---|
| Network / 5xx from Gemini (1 retry, 5 s timeout) | `200` with `comment:"⚠ 醫師建議產生失敗：<reason>"` and the disclaimer | Heads & user_message still returned — POC stays useful when only LLM is down |
| Gemini empty / safety-blocked | Same as above with reason `"模型未產生回應"` | |
| ADK auth missing / expired | `503 {error:"llm unavailable"}` on `/api/analyze`; `/health` exposes a status badge | |

### Config errors (block the write, never the runtime)

| Case | Response |
|---|---|
| YAML parse fails on `PUT /api/config/{section}` | `422 {error:"yaml parse error", line:N, message:…}` |
| Registry: required key missing / `onnx_path` not on disk | `422 {error:"registry validation", failures:[{task, reason}, …]}` |
| LLM: `temperature` outside [0,2], etc. | `422 {error:"invalid value", field:"temperature"}` |
| `POST /api/config/registry/reload` and 1 of N ONNX fails to load | `200 {loaded:[…], failed:[{task, reason}]}` — partial reload kept; if **zero** heads load, registry rolls back to previous good state |

### Lifecycle / cold start

- On startup, if `system.current.md` / `*.current.yaml` are missing, copy from the corresponding `*.default.*`.
- Registry loaded synchronously at startup. If it fails entirely, server still boots — `/api/analyze` returns `503`, registry tab shows the error so the user can fix the YAML and reload.
- Concurrency: single-process uvicorn for POC. Last-write-wins on config files; not worth locking.

## 10. Testing

POC-appropriate, not coverage theatre. Tests document each boundary's contract; mock anything slow / network / expensive; skip Gradio UI rendering tests (covered by manual smoke).

### `tongue-ai`

| Module | Tests |
|---|---|
| `preprocessing` | resize keeps aspect; normalise produces `~N(0,1)` for ImageNet; BGR↔RGB round-trip is identity |
| `task_head` | stub `ort.InferenceSession` returning fixed logits → `predict()` decodes to expected `HeadResult`; multi-label threshold cuts correctly |
| `registry` | valid YAML → expected `TaskHead` count; missing `onnx_path` raises; missing required key raises with field name |
| `inference.run_all` | one head raising doesn't break the loop; output ordering matches registry order |
| `detection` | when no detector configured, returns whole-image bbox |

### `tongue-backend` (integration via FastAPI `TestClient`)

| Surface | Tests |
|---|---|
| `pipeline.analyze` | stub registry + stub LLM client → asserted response shape; LLM failure produces error-stamped comment |
| `POST /api/analyze` | 200 with valid jpg fixture; 400 missing file; 400 corrupt bytes; 413 oversize |
| `POST /api/analyze` (LLM down) | 200 with `comment` containing failure stamp; heads still populated |
| `GET/PUT/POST /api/config/{section}` | round-trip persistence, reset restores default, malformed YAML → 422 with line number |
| `POST /api/config/registry/reload` | partial-fail keeps loaded heads; total-fail rolls back to previous registry |
| `llm/user_message.build` | golden tests: known `HeadResult[]` → exact zh-TW bullet list |
| `llm/prompt_store` | first-boot copies default; `reset()` overwrites current |
| `llm/client` | mocked at the SDK boundary — assert ADK is called with `(system, user, llm_config)` |

### `tongue-frontend`

| Surface | Tests |
|---|---|
| `api.py` | each function hits the right URL/method/payload — verified with `respx` mock transport |
| Gradio views | **no automated tests** — covered by smoke script |

### Tooling

`pytest`, `pytest-asyncio`, `httpx[test]`, `respx`, `pyyaml`. No new heavy deps beyond what the runtime already pulls in.

### Not tested in POC

- Gradio UI rendering or `gradio_client` interaction flows.
- Real Gemini calls in CI (mocked at the SDK boundary; exercised by manual smoke).
- ONNX numerical accuracy (the models' contract, not the harness's).
- Load / concurrency / GPU paths.
- TLS / CORS edge cases.

## 11. Open assumptions to verify during implementation

These are assumptions baked into the design that should be confirmed once a real model is wired in:

1. **Tongue ROI detection** — design assumes an optional ONNX detector. If actual task models accept full images and were trained on raw photos, drop the detection step (registry's `detector:` block null).
2. **Per-task preprocessing** — registry allows each task its own input size and normalisation. If all task models share preprocessing, this is over-flexible but harmless.
3. **Multi-label threshold** — defaults to 0.5 per head; tune per task in YAML.
4. **Gemini via Vertex AI** through `google-adk`. If the project uses the direct Gemini API (`google-genai`) instead, swap the client wrapper; the rest is unchanged.
5. **Class name encoding** — registry's `class_names` are zh-TW strings matching the v4 schema verbatim. The user message and Gemini prompt depend on this.

## 12. Out of scope (named, not sneaked in later)

- Authentication / authorisation on any endpoint.
- Audit log of config changes.
- Rate limiting on `/api/analyze`.
- Persistent session history.
- Multimodal LLM fallback (sending image to Gemini if classifiers fail).
- Streaming partial results (SSE).
- Structured config forms (raw YAML/text only for POC).
